9. TECHNICAL IMPLEMENTATION DETAILS
================================================================================

SCROLL PRESERVATION ALGORITHM:
When displaying more messages, prevent scroll jump:
1. Store current scroll height: prevScrollHeight = container.scrollHeight
2. Increase displayLimit, trigger re-render
3. Wait for render: requestAnimationFrame()
4. Calculate new scroll: scrollHeight - prevScrollHeight
5. Set: container.scrollTop = scrollHeight - prevScrollHeight
6. User stays at same visual position despite DOM changes above

WHY SSE (Server-Sent Events) FOR STREAMING:
- Browser built-in EventSource API (no library needed)
- Unidirectional server→client (simpler than WebSocket)
- Stateless HTTP-based (no persistent connection issues)
- Browser auto-reconnection
- Ideal for streaming responses

DISPLAY CALCULATION:
messages.slice(-displayLimit)
- Negative index: count from end
- slice(-50): Last 50 elements
- slice(-100): Last 100 elements
- Efficient: No array copy, just view

STATE UPDATES PATTERNS:
1. Replace: setMessages(parsed) - Full reload
2. Append: setMessages(prev => [...prev, newMsg]) - Add to end
3. Slice: messages.slice(-displayLimit) - Calculate display
4. Clear: setMessages([]) - Empty

10. PERFORMANCE CHARACTERISTICS
================================================================================

MEMORY USAGE:
- Stores ALL messages in state (no cleanup)
- Risk: Large conversations consume significant memory
- Optimization needed: Virtual scrolling or server pagination

RE-RENDER TRIGGERS:
1. Message load: Full component re-render
2. Each scroll-to-top: Display slicing only (minimal)
3. Each stream chunk: Full re-render
   - Typical: 10-50 renders per response
   - Could optimize with useMemo or separate component

NETWORK REQUESTS PER MESSAGE:
1. GET /threads/{id}/messages/stream/ (persistent SSE)
2. GET /threads/ (refresh threads)
3. GET /threads/{id}/messages/ (reload after stream)
4. Optional PATCH /threads/{id}/ (rename if new thread)
Total: 2-4 requests per message send
